# 通用投资视频分析

## 定位

本 skill 只用于其他作者专属 skill 尚未覆盖的投资类普通公开视频，作者、市场和板块均不固定。目标是把视频转成可追溯的学习资料：先形成覆盖全片的可读文字稿，再基于文字稿生成分析总结。

## 作者路由规则

开始处理前，必须先识别作者，并检查当前可用的作者专属 skill。作者专属 skill 的优先级高于本 skill：

- “比巴卜作手”相关视频 → 使用 `analyze-bibabu-zuoshou`。
- “数字K哥”相关视频 → 使用 `analyze-shuzi-kge`。
- “袁帅马”相关视频 → 使用 `analyze-yuanshuaima`。
- “视野环球财经”或 “RhinoFinance” 相关视频 → 使用 `analyze-rhino-finance`。
- “小翠时政财经”相关视频 → 使用 `analyze-xiaocui-news`。

如果视频作者属于上述作者，即使用户只要求普通视频总结，也必须转交相应作者专属 skill；如果用户同时提供多个作者的视频，应按作者分别路由。只有在作者不属于上述范围，或当前已有作者专属 skill 未覆盖该作者时，才使用本 skill。作者无法确认时，先根据标题、频道名、页面信息和视频内容判断；仍无法确认时，按未知作者处理并在输出中注明。

默认归档到 `docs/trading/video-analysis/`。若用户明确要求沉淀为某位作者的人物专题，再改用或新建 `docs/trading/experts/<slug>/`。

## 工作流

1. 识别视频信息：标题、作者、平台、发布日期、时长、链接、可见性、字幕情况和主要市场。
2. 先做文字稿：优先使用视频自带字幕；字幕缺失或质量不足时转写音频。整理时保留作者表达顺序、语气强弱、转折、例子和有信息量的重复。
3. 核实名词：对视频提到的股票、ETF、指数、期货、币种和重要专有名词逐一核对。无法确认时保留时间点和上下文，标记“待核”，不要凭发音硬猜。
4. 冻结文字稿：确认全片有效议题、重点论证、全部标的和风险提示均已进入正式文字稿后，再写总结。
5. 生成总结：以冻结文字稿为主要依据，按视频本身的结构自由组织，不强行套固定作者模板。
6. 发布检查：更新目录、索引和站点导航；必要时运行项目校验命令。临时视频、音频、原始转写和截图候选文件只放临时目录，任务结束前逐个明确路径删除。

删除文件时只能一次删除一个明确路径的文件，禁止批量删除目录或使用递归删除命令。

## 视频处理提示

- 执行过程中涉及 Python 脚本、`yt-dlp`、字幕处理、音频下载、语音转写、截图、图片处理或校验命令时，优先使用本项目本地 `uv` 环境运行，例如 `uv run ...` 或 `uv run python <script>`；只有本地环境缺少能力或命令失败且确认无法修复时，才临时改用系统环境，并在最终说明中标注原因。
- 平台识别后选择最稳路径：B 站和 YouTube 优先使用用户当前浏览器的登录态读取视频元信息、字幕列表、章节、简介、播放地址和可见性信息；可通过已登录的浏览器会话或导出的 cookies 访问。公开接口、`yt-dlp` 或匿名请求只作为补充路径，用于交叉核验或在登录态不可用时兜底。本地视频则先从文件名、用户说明和媒体元数据建立基本信息。
- 字幕优先级：先检查平台是否提供创作者字幕或 AI/自动字幕；有字幕时优先下载字幕并评估质量，质量可用就直接基于字幕整理文字稿。只有在没有字幕、字幕不可访问、字幕时间轴严重缺失或内容明显错漏时，才下载音频并本地转写。字幕和转写都不完整时，不伪造作者原话，使用“待补录/待核”标记。
- 需要下载音频时，优先用项目脚本转写：`uv run python scripts/transcribe_audio.py <audio-path> -o <raw-transcript-path> --wav <wav-path> --model small --language zh --print`。该脚本负责调用 `ffmpeg` 转为 16 kHz 单声道 wav，并用 `faster_whisper` 生成带时间戳的过程底稿；若脚本失败，再根据错误修复脚本或临时改用等效命令，并在最终说明中标注。
- 文字稿制作分两步：先保留原始字幕或转写结果作为过程底稿，再整理成可读时间轴文字稿。原始底稿只放临时目录，不提交到知识库。
- 需要截图时，用 PyAV 或等效工具按时间点精确抽帧，Pillow 或等效工具输出 JPEG。优先截取图表、板块对比、标的列表、作者标注、交易计划和风险条件，避免只截口播画面。
- 截图数量按内容决定：一般视频可选 4 至 8 张；如果视频主要是口播或画面信息价值低，可以少截或不截，并在总结中说明依据主要来自文字稿。
- 标的核名优先级为“清晰画面 > 官方字幕或创作者字幕 > 视频上下文 > 自动转写/语音识别”。仍不确定时保留时间点和上下文，不静默删除。
- 不把下载的视频、音频、原始字幕、自动转写中间稿、候选帧或联系表提交到仓库。正式产物只保留整理后的文字稿、总结和经过选择的截图。

## 输出文件

每期默认产出两个文件：

- `docs/trading/video-analysis/transcripts/YYYY-MM-DD-<slug>.md`
- `docs/trading/video-analysis/reviews/YYYY-MM-DD-<slug>.md`

若同一天有多个视频，用标题、平台 ID 或作者拼音补充 `<slug>`。截图如需保存，放到：

- `docs/public/images/trading/video-analysis/YYYY-MM-DD-<slug>/`

新增文章后同步维护：

- `docs/trading/video-analysis/index.md`
- `docs/trading/video-analysis/transcripts/index.md`
- `docs/trading/video-analysis/reviews/index.md`
- `docs/.vitepress/config.mts` 顶部导航里的“视频分析”入口：在“分析总结”和“视频文字稿”下补新增文章链接。
- `docs/.vitepress/config.mts` 侧边栏里的“视频分析”目录：若目录不存在则新建；若已存在则按日期倒序补“分析总结”和“视频文字稿”的新增文章链接。

发布前必须用 `rg` 检查新文章 slug 是否同时出现在正式文件、索引页和 `docs/.vitepress/config.mts` 中；不能只更新页面内索引而漏掉站点目录。

## 文字稿要求

文字稿服务于追溯，不是摘要。建议结构：

- 视频信息：标题、作者、平台、日期、时长、原链接、字幕或转写来源、准确性说明。
- 时间轴文字稿：按自然议题分段，每段带起始时间。行情切换、板块切换、个股切换和观点转折处增加时间点。
- 全部标的提及表：名称、代码或交易所、类型、出现时间、作者说法、核验状态。
- 待核清单：无法确认的名称、时间点和上下文。

整理文字稿可以轻度去除口头填充词，但不要把作者口语改成分析师摘要；不要补写作者没说过的因果链。

## 总结要求

总结优先忠实呈现视频逻辑，再做学习化整理。可按视频内容自由选择以下模块：

- 核心结论：只写作者明确表达或由其上下文直接推出的观点。
- 市场环境：指数、流动性、情绪、风险偏好、宏观或消息背景。
- 板块与主题：强弱排序、催化因素、持续性条件、分歧点。
- 标的梳理：核心标的、跟风标的、观察标的、回避标的，以及作者给出的条件。
- 交易计划：参与前提、观察信号、仓位态度、止损或失效条件。
- 风险提示：视频里的风险、AI 整理时发现的表达不确定处、待验证信息。
- 可迁移知识：方法论、判断框架、常见误区和适用边界。延展内容必须标注为“整理者延展”。

不要把“可以观察”改成“明确买入”，不要把短线交易计划改成长期投资结论。所有买卖相关内容都写成条件句，并保留“非投资建议”声明。

## 核验边界

必须核验：

- 视频元数据。
- 标的名称、代码、交易所和类型。
- 作者提到且影响理解的明显事实，例如财报日期、重大新闻、指数点位或已发生事件。

谨慎处理：

- 作者的主观判断、交易偏好、技术形态解释和未来预测，只按视频语境归纳，不用外部资料替作者背书。
- 若外部资料与视频说法冲突，在总结中标注“视频说法”和“外部核验”差异，不静默改写。

## 临时文件清理

- 创建临时文件时同步记录明确路径，只清理本次任务生成的文件。
- 清理放在工作流最后：先确认文字稿、总结、截图、索引和侧边栏均已落盘并完成必要检查，再删除临时文件。
- 临时目录中含有多个文件时，逐个核对并逐个删除；不得直接删除整个目录。若必须批量删除，应停止并请求用户手动处理。
- 最终回复中简要说明临时文件是否已清理；若某个临时文件因占用、权限或用途未确认而保留，列出明确路径和原因。

## 风格

保持学习笔记口吻：清楚、可追溯、有条件、有边界。允许 AI 根据不同市场和视频类型自由调整结构，但必须做到先文字稿后总结、标的全量核名、观点与延展分开。
