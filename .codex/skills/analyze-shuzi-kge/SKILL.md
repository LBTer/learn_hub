# 数字K哥视频分析

## 执行前必读

数字K哥 B 站主页：https://space.bilibili.com/3493268855720591

本 skill 只处理公开普通视频，不设置会员视频、会员专题或付费内容分流。每期固定产出两个板块：

- 日常分析：基于冻结文字稿生成的图文总结。
- 视频文字稿：覆盖全片有效议题的可读时间轴文字稿。

执行过程中涉及 Python 脚本、下载、转写、截图、图片处理或校验命令时，明确优先使用本项目本地 `uv` 环境运行，例如 `uv run ...`；只有本地 `uv` 环境缺少能力或命令失败且确认无法修复时，才临时改用系统环境，并在最终说明中标注原因。

当用户要求检查数字K哥当天是否有新视频时，优先使用本 skill 内的 `scripts/check-new-videos.py`。该脚本默认检查北京时间当天，临时读取本机 Chrome 登录态，不导出或保存 Cookie；如需回看指定日期，传入 `--date YYYY-MM-DD`。若最近投稿列表中个别历史视频元数据读取失败，脚本会在 stderr 中告警并跳过该条，继续判断其他视频，避免单条异常中断当天检查。

## 类型识别与路由

1. 获取标题、UP 主、发布日期、时长、原始链接、分 P、`cid`、字幕列表和页面可见性。
2. 确认视频属于数字K哥公开视频。账号名以 B 站页面为准；若用户给的是合集、动态、直播回放或其他 UP 主转载，先说明识别结果，再按实际可获取内容处理。
3. 不做会员分类。即使标题出现课程、训练、答疑、复盘等字样，只要是数字K哥公开视频，仍按“视频文字稿 + 日常分析”处理。
4. 每期必须先完成并冻结视频文字稿，再生成日常分析。

## 日常分析工作流

1. 核验视频标题、UP 主、发布日期、时长和原始链接。
2. 必须严格串行执行以下阶段，不得交叉生成文字稿和总结：
   - 阶段一：文字稿提取与整理。获取字幕；无字幕时转写音频。先形成忠实保留识别内容的原始提取稿，再以原稿为底稿制作覆盖全片的可读时间轴文字稿。此阶段不写总结。
   - 阶段二：股票名核实。以整理文字稿和“本期全部股票提及表”为依据，结合清晰画面、字幕、上下文及必要的公开行情信息，逐一核实视频提到的全部股票名称；无法确认时保留时间点和上下文并标记“待核”。
   - 阶段门：文字稿冻结。确认全片有效议题、全部股票、长篇观点、教学内容和后续计划均已纳入，且“本期全部股票提及表”已完成。将文字稿完整写入正式文件后冻结。
   - 阶段三：日常分析生成。只能在文字稿冻结后创建或填写日常分析，并以冻结文字稿为唯一内容底稿。
   - 阶段四：发布检查。完成关键帧、交叉检查、索引与侧边栏更新、构建校验和临时文件清理。
3. 内容核实仅限于文字稿阶段的股票名称核实。除视频元数据确认外，不要求对题材逻辑、市场数据、主播判断或新闻背景做额外事实核验；不得把外部信息补写成主播观点。
4. 在文字稿冻结前禁止创建或预写日常分析正文。日常分析不得直接基于零散字幕、音频片段、原始提取稿、标题、截图或记忆生成；冻结后若文字稿发生实质修订，必须重新检查并同步修订日常分析。
5. 按以下顺序梳理内容：
   - 市场环境：指数位置、量能、情绪、赚钱效应和风险状态。
   - 主线与板块：逐个板块记录作者看法、强弱排序、资金态度、催化因素和后续观察点。
   - 具体个股：逐板块列出视频明确提到的每一只股票，说明作者语境中的核心、中军、辨识度、补涨、跟风、竞争关系和失效条件。
   - 操作计划：观察条件、参与前提、仓位倾向、卖点或止损、后续预案。
   - 心态纪律：作者明确表达的风险控制原则、等待条件和回避场景。
   - 篇幅较长的观点：无论属于上述哪个部分，都要按作者原有展开顺序充分保留论据、推演、例子、转折、条件和结论；不得压缩成一句结论。
6. 选择足以覆盖市场总览和主要板块的关键截图，通常为 6 至 10 张。优先截取图表、计划表、板块对比、重点个股和失效条件，避免只截口播画面。
7. 每张图注明原视频时间点。解释优先沿用作者表达顺序、判断措辞和语气，再补充必要的结构化说明；截图仅作学习引用，不替代原视频。
8. 截图只放入“日常分析”正文。视频文字稿仅保留时间轴文字和“本期全部股票提及表”，不得插入截图或图片引用。
9. 日常分析固定分为“每日信息”和“知识点”两个一级部分。知识点中的延展内容必须说明是分析者推导，不冒充主播原话。
10. 输出到人物专题时，采用“人物主页 → 日常分析 / 视频文字稿 / 技术点学习 → 日期文章”的目录层级。
11. “每日信息”固定包含三个部分：
    - 第一部分“市场总览”：指数、赚钱效应、资金结构和一句话结论。
    - 第二部分“具体分析”：先给出板块观点速览表，再逐板块整理作者观点、重点个股、截图依据和后续关注。
    - 第三部分“后续计划”：指数预案、板块预案、仓位动作、失效条件和观察清单。
12. “知识点”记录当日可迁移方法、延展知识、适用条件和失效条件。
13. 独立的“技术点学习”模块用于积累多个交易日后做跨期归纳。除非用户明确要求归纳或更新，否则只保留模块入口，不新增日期文章、不搬运单日知识点。
14. 每期必须在本项目中产出两个文件，固定名称为“日常分析”和“视频文字稿”，不得使用其他模块名：
    - `docs/trading/experts/shuzi-kge/transcripts/YYYY-MM-DD.md`：覆盖全片议题的可读时间轴文字稿。
    - `docs/trading/experts/shuzi-kge/daily-analysis/YYYY-MM-DD.md`：基于完整文字稿制作的图文分析总结。
15. 保留原视频链接、项目内文字稿链接与“非投资建议”声明，并同步更新人物主页、日常分析索引、文字稿索引和 VitePress 侧边栏。每新增、改名或迁移一期文章，都必须检查 `docs/.vitepress/config.mts` 中“数字K哥”目录，按日期倒序新增或修正对应的日常分析、视频文字稿入口；不得因模块索引已更新而跳过侧边栏目录更新。
16. 确认项目文字稿、总结和关键帧全部落盘后，逐个删除本期在 `/tmp` 生成的视频、音频、原始自动转写、候选帧和联系表，不保留本地临时副本。删除文件时只能一次删除一个明确路径的文件，不得批量删除目录或使用递归删除命令。

## 工具选择原则

- 优先使用本项目本地 `uv` 环境执行所有 Python 相关工具和脚本，包括 `yt-dlp`、`faster-whisper`、PyAV、Pillow、截图处理、`quick_validate.py` 以及临时辅助脚本；命令形式优先为 `uv run <command>` 或 `uv run python <script>`。
- 首选 B 站公开接口读取标题、UP 主、发布时间、时长、`cid`、字幕列表和播放地址。
- 公开接口缺失字幕或需要登录权限时，在用户明确授权后可使用 Chrome 中现有的 B 站登录态继续获取；即使已获登录态授权，也应优先尝试接口方式读取元数据、字幕和播放地址。
- 首选视频自带字幕；字幕缺失或质量不足时使用 `faster-whisper` 转写。
- 使用 PyAV 精确定位时间点，Pillow 输出 JPEG 和候选帧联系表，再人工复核图片。
- 使用 `rg`、`sed` 检索转写稿，先建立观点时间轴和全部股票提及清单，再整理并冻结文字稿，最后生成日常分析。
- 股票名按“清晰画面 > 官方元数据或字幕 > 上下文 > 语音识别”核实；题材名及其他内容不做独立核实，按文字稿原始语境保留。
- 使用 `quick_validate.py`、站点构建命令和 `git diff --check` 做发布前检查。
- 原视频、音频和未经整理的自动转写只允许暂存在临时目录，不提交到仓库。整理后的可读时间轴文字稿必须写入本项目，任务结束前逐个清除本期全部临时文件。

## 文字稿规范

- 文字稿制作分为两个连续步骤，不得跳过原始提取稿直接写结构化文字稿或总结：
  1. 原始提取稿：直接落下字幕或语音识别结果，尽量保持作者原始顺序、说法、语气强弱、重复、犹豫、自我修正、类比和例子，不先概括、改写、重组、核名替换或补充分析。仅可补充时间点、说话人和无法识别标记。
  2. 整理文字稿：必须在原始提取稿上逐段处理，划分自然段和议题，做不改变原意与语气强弱的轻度文字润色，删除纯填充语气词和无意义重复，并结合画面与上下文修订明显识别错误；外部核实只用于确认股票名称，不核实题材名或其他内容。
- 原始提取稿是过程性底稿，可暂存在临时目录，不要求提交；项目中的视频文字稿为第二步完成的整理稿。
- 覆盖从开场到收尾的全部有效议题，不只摘录高光、结论或代表性股票。
- 每个自然议题标注起始时间，行情切换、板块切换、个股切换和观点转折处必须增加时间点。
- 整理时不得把口语改写成摘要或分析结论；最大化保留作者论述顺序、常用说法、语气强弱、犹豫、自我修正、类比、例子和有信息量的重复。
- 作者对某一观点讲述较多、反复补充或连续推演时，整理文字稿必须给予与原视频相称的篇幅，完整保留问题背景、判断依据、推理过程、举例、反例、条件限制、观点转折和最终落点。
- 自动转写不能直接标成逐字原话。明显识别错误按画面和上下文修正，无法确认的股票或专有名词保留时间点并标记“待核”。
- 单独维护“本期全部股票提及表”，记录股票名、代码（能确认时）、所属板块、出现时间、作者说法和核验状态。
- 同一股票多次出现时，不只保留第一次；应合并列出关键时间点和观点变化。
- 开头注明视频信息、转写来源、视频时长和准确性限制，并链接回原视频。
- 只有股票名核实完成、覆盖度检查通过并写入正式文件的整理文字稿才算“冻结”。截图选择和日常分析必须以冻结文字稿为依据。

## 分析准则

- 将预测改写为条件句，例如“若次日放量站稳压力位，才提高继续修复概率”。
- 区分板块“涨得多”和“有持续性”。持续性至少考察逆势抗跌、修复主动、核心带动、量价健康。
- 区分消息刺激和内生趋势。消息日的一致高潮不等于可追买。
- 记录否证条件。没有退出条件的看多观点不算完整交易计划。
- 对个股仅复述视频语境，不提供确定性买卖指令。
- 板块速览表至少包含“板块、作者观点、全部提及个股、后续关注”四列。
- 视频明确提及的股票必须全部进入文字稿和“全部股票提及表”，不能因为不是核心、只出现一次或观点偏负面而省略。
- 股票名无法确认时保留其时间点、上下文和“待核”标记，不凭发音猜测，也不因不确定而静默删除。
- 总结优先忠实呈现作者的原始表达逻辑。结构化改写不得改变语气强弱，例如不能把“可以观察”改成“明确看好”。
- 对每个主要板块说明其定位：主线候选、消息修复、高低切、量化轮动、情绪 PK、避险或回避。
- 如转写与画面冲突，以画面和上下文为准；仍不确定时使用“疑似”或省略名称。

### 人物主页

只维护模块入口和说明，不堆放日期归档或单期正文。默认入口包括“日常分析”“视频文字稿”“技术点学习”。

### 日常分析模块

在 `daily-analysis/index.md` 按日期倒序维护文章入口。每篇日期文章固定包含：

- `## 每日信息`
  - `### 第一部分：市场总览`
  - `### 第二部分：具体分析`
  - `### 第三部分：后续计划`
- `## 知识点`：当日方法提炼、适用条件、常见误区、失效条件和交易准则，明确标注分析与延展。

固定一级、二级结构不限制正文展开。凡作者在视频中重点长篇讨论的内容，可在对应部分下增加四级标题、加粗小标题或多个独立小点，篇幅以完整保留观点为优先，不以追求短摘要为目标。

### 文字稿模块

在本项目的 `docs/trading/experts/shuzi-kge/transcripts/` 保存文字稿，并在 `transcripts/index.md` 按日期倒序维护入口。每篇文字稿包含视频信息、准确性说明、全片时间轴和“本期全部股票提及表”。文字稿服务于追溯作者完整表达，不替代原视频，也不把自动识别文本冒充逐字稿。

`daily-analysis/index.md` 的模块标题固定为“日常分析”，`transcripts/index.md` 的模块标题固定为“视频文字稿”；日期文章标题分别使用 `YYYY-MM-DD 日常分析` 和 `YYYY-MM-DD 视频文字稿`。

### 站点目录同步

每期文档落盘后必须将文章加入 `docs/.vitepress/config.mts` 的“数字K哥”侧边栏目录，并与模块索引保持一致：

- 同时加入“日常分析”和“视频文字稿”的日期列表。
- 日期纠正、文件改名或路径迁移时，必须同步替换旧目录链接，不能保留失效入口。
- 发布前使用 `rg` 对照文章路径、模块索引和 VitePress 配置，确认三处均已收录且日期倒序正确；站点构建成功不代表目录已经更新，必须单独核对。

### 技术点学习模块

`technical-notes/index.md` 是未来跨期归纳入口。默认不从单日视频直接更新；积累足够样本并经用户明确要求后，再按主题总结稳定方法、常见偏差及适用边界。

普通视频截图保存到 `docs/public/images/trading/shuzi-kge/YYYY-MM-DD/`。文件名优先使用四位秒数，便于追溯原视频时间点。

## 专属观察框架

读取 [references/expert-profile.md](references/expert-profile.md)，持续维护数字K哥的稳定方法、常用指标、题材分类方式、风控语言和历史观点命中情况。每期只在有新证据时更新画像，不因单条视频下结论。
